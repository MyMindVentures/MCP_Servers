## MCP Server Template (Node.js / TypeScript)

This repository contains a reusable Node.js/TypeScript MCP Server Template that can be configured per upstream app (Raindrop, Notion, Airtable, etc.).

### Features

- **MCP transports**: HTTP `/mcp` (streamable) and `/sse` (SSE).
- **Capability registries**: tools, resources, prompts, optional sampling.
- **Execution engine**: HTTP client with timeouts, retries, and normalization.
- **Security**: API key / JWT hooks, upstream allowlists.
- **Observability**: structured logging, metrics, tracing hooks.
- **OpenAI compatibility**: `/openai/tools`, `/openai/execute` façade.

### Quick start

1. Install dependencies:

```bash
npm install
```

2. Run in development:

```bash
npm run dev
```

3. Build and run:

```bash
npm run build
npm start
```

Configuration lives in `config/` and can be overridden via environment variables (see `.env.example`).

# Parallax MCP Server Template

Turn any API into an MCP‑ready, enterprise‑friendly toolset in minutes. This template gives you a beautiful admin UI for no‑coders, a powerful dynamic tool engine for developers, and a secure HTTP endpoint that LLMs can call from anywhere.

- ✅ **No‑code workflow builder** at `/admin/settings` – pick a platform like Airtable, paste an API key, and click **Discover endpoints**.
- ✅ **Dynamic MCP tools from OpenAPI** – automatically generate tools from your API spec or a well‑known SaaS platform.
- ✅ **Multi‑transport support** – stdio (Claude Desktop, Postman) and Streamable HTTP `/mcp` (ChatGPT / OpenAI Agents, hosted LLMs).
- ✅ **Enterprise‑ready** – run in your own cloud, fronted by HTTPS and your preferred auth (bearer token or upstream OAuth/JWT proxy).

Under the hood, this project is generated by the [Postman MCP Generator](https://postman.com/explore/mcp-generator) and implements the [Model Context Protocol (MCP)](https://modelcontextprotocol.io/introduction).

## Connection patterns (MCP clients)

### Local (stdio) – Claude Desktop, Postman MCP

- Run the server locally:

```sh
npm install
npm run start:stdio
```

- Point your MCP client at `node /absolute/path/to/mcpServer.js` as described below in the original generator instructions.

### Remote HTTP – `/mcp` for ChatGPT / OpenAI Agents and others

When you run:

```sh
npm run start:http
```

the server exposes a Streamable HTTP MCP endpoint at:

- `http://localhost:3001/mcp` by default, or
- `https://your-domain.example.com/mcp` when deployed behind HTTPS.

Recommended env vars for production:

```sh
# Public base URL your LLM platform should call
PUBLIC_MCP_BASE_URL=https://your-domain.example.com

# Optional static bearer token to protect /mcp and /health
MCP_SERVER_TOKEN=super-secret-mcp-token
```

Example protected MCP call:

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $MCP_SERVER_TOKEN" \
  "$PUBLIC_MCP_BASE_URL/mcp" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
```

See `docs/deployment-and-security.md` for reverse‑proxy and OAuth/JWT integration patterns.

### Dual mode (local + public)

Run **one process** with both stdio and Streamable HTTP so you can use local clients (e.g. Claude Desktop, Postman) and public HTTP (e.g. ChatGPT via ngrok) at the same time:

```sh
npm run start:dual
```

or:

```sh
node mcpServer.js --dual
```

The HTTP endpoint is on `PORT` (default 3001) at `/mcp`; stdio is available to the same process for local MCP clients. You cannot use `--dual` with `--catalog-only` or `--sse`.

### Development with Docker (live preview + hot reload)

To run the app in Docker with **hot reload** so that edits to `public/`, `lib/`, `routes/`, or `mcpServer.js` are picked up without rebuilding:

```sh
docker compose up --build
```

Then open **Splash:** http://localhost:3002 and **Home:** http://localhost:3002/home.

The compose setup mounts the project into the container (so file changes are visible), runs **nodemon** with `--streamable-http` so the server restarts on changes, and keeps container `node_modules` so the install stays consistent. The host port defaults to **3002** to avoid conflicting with a local server on 3001; set `COMPOSE_HOST_PORT=3001` in `.env` or in the shell if you want to use 3001.

**Separate catalog service:** Compose also starts a **catalog** container (API + DB + scheduler) on host port **3003**. The app container uses `CATALOG_SERVICE_URL=http://catalog:3001` so it talks to the catalog over the Docker network. Catalog uses PostgreSQL by default (a postgres service runs in Compose). Set POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB in .env to override. Catalog data lives in Postgres; the catalog_data volume is for snapshot storage. For “From catalog” discovery in the admin, choose a provider from the dropdown; the catalog base URL can stay empty when using Docker (the app uses the catalog service automatically).

## 🚦 Getting Started

### ⚙️ Prerequisites

Before starting, please ensure you have:

- [Node.js (v18+ required, v20+ recommended)](https://nodejs.org/)
- [npm](https://www.npmjs.com/) (included with Node)

This MCP server uses [postman-runtime](https://www.npmjs.com/package/postman-runtime) to execute requests, ensuring full compatibility with all Postman features including complex authentication types.

### 📥 Installation & Setup

**1. Install dependencies**

Run from your project's root directory:

```sh
npm install
```

### 🔐 Set tool environment variables

In the `.env` file, you'll see environment variable placeholders, one for each workspace that the selected tools are from. For example, if you selected requests from 2 workspaces, e.g. Acme and Widgets, you'll see two placeholders:

```
ACME_API_KEY=
WIDGETS_API_KEY=
```

Update the values with actual API keys for each API. These environment variables are used inside of the generated tools to set the API key for each request. You can inspect a file in the `tools` directory to see how it works.

```javascript
// environment variables are used inside of each tool file
const apiKey = process.env.ACME_API_KEY;
```

**Caveat:** This may not be correct for every API. The generation logic is relatively simple - for each workspace, we create an environment variable with the same name as the workspace slug, and then use that environment variable in each tool file that belongs to that workspace. If this isn't the right behavior for your chosen API, no problem! You can manually update anything in the `.env` file or tool files to accurately reflect the API's method of authentication.

## 🌐 Test the MCP Server with Postman

The MCP Server (`mcpServer.js`) exposes your automated API tools to MCP-compatible clients, such as Claude Desktop or the Postman Desktop Application. We recommend that you test the server with Postman first and then move on to using it with an LLM.

The Postman Desktop Application is the easiest way to run and test MCP servers. Testing the downloaded server first is optional but recommended.

**Step 1**: Download the latest Postman Desktop Application from [https://www.postman.com/downloads/](https://www.postman.com/downloads/).

**Step 2**: Read out the documentation article [here](https://learning.postman.com/docs/postman-ai-agent-builder/mcp-requests/create/) and see how to create an MCP request inside the Postman app.

**Step 3**: Set the type of the MCP request to `STDIO` and set the command to `node </absolute/path/to/mcpServer.js>`. If you have issues with using only `node` (e.g. an old version is used), supply an absolute path instead to a node version 18+. You can get the full path to node by running:

```sh
which node
```

To check the node version, run:

```sh
node --version
```

To get the absolute path to `mcpServer.js`, run:

```sh
realpath mcpServer.js
```

Use the node command followed by the full path to `mcpServer.js` as the command for your new Postman MCP Request. Then click the **Connect** button. You should see a list of tools that you selected before generating the server. You can test that each tool works here before connecting the MCP server to an LLM.

## 👩‍💻 Connect the MCP Server to Claude

You can connect your MCP server to any MCP client. Here we provide instructions for connecting it to Claude Desktop.

**Step 1**: Note the full path to node and the `mcpServer.js` from the previous step.

**Step 2**. Open Claude Desktop → **Settings** → **Developers** → **Edit Config** and add a new MCP server:

```json
{
  "mcpServers": {
    "<server_name>": {
      "command": "</absolute/path/to/node>",
      "args": ["</absolute/path/to/mcpServer.js>"]
    }
  }
}
```

Restart Claude Desktop to activate this change. Make sure the new MCP is turned on and has a green circle next to it. If so, you're ready to begin a chat session that can use the tools you've connected.

**Warning**: If you don't supply an absolute path to a `node` version that is v18+, Claude (and other MCP clients) may fall back to another `node` version on the system. Make sure to use Node.js v18 or higher for best compatibility.

## 🤖 Connect the MCP Server to ChatGPT / OpenAI (HTTP)

This server already supports the recommended MCP transports for OpenAI:

- **Stdio** (default): `node mcpServer.js` or `npm run start:stdio`
- **Streamable HTTP** (recommended for remote / cloud): `node mcpServer.js --streamable-http` or `npm run start:http`

When you run with `--streamable-http`, the server exposes a `/mcp` endpoint on `PORT` (default `3001`). For production use with ChatGPT / OpenAI Agents:

- Run behind HTTPS (via your own reverse proxy / load balancer)
- Point your OpenAI app / agent configuration at `https://your-domain.example.com/mcp`
- Provide any required environment variables (API keys, base URLs) via your deployment platform

**Quick test from ChatGPT on your tablet:** Use Docker + ngrok and add the MCP app in ChatGPT with the public URL. See **[Test with ChatGPT (tablet)](docs/chatgpt-quickstart.md)** for step-by-step instructions.

> Note: SSE (`--sse`) is available for legacy clients but Streamable HTTP and stdio are preferred for new integrations.

## 🧩 Using this as an Enterprise Template

This project can act as a reusable template for **per‑platform MCP servers** (e.g. Airtable, Notion, Raindrop):

- **Per app**: create a new copy of this project and drop in Postman‑generated tools for that API under `tools/`
- **Tool registration**: reference those tools from `tools/paths.js`
- **Environment**: define API keys / base URLs in `.env` and/or Postman collection variables
- **Transports**: choose `stdio` for desktop clients (Claude, Postman) or `--streamable-http` for remote clients (ChatGPT / OpenAI Agents)

Each per‑app server is then built as its own Docker image and deployed as a separate container, giving you isolated, full‑control MCP servers per platform.

## 🛡️ Admin UI & Configuration

This template now includes a lightweight admin UI and config backend so you can turn any OpenAPI‑described API into an MCP‑compatible toolset without touching code.

### Initial admin credentials

- **Username**: from `ADMIN_USERNAME` (default: `admin`)
- **Password**: from `ADMIN_PASSWORD` (default: `change_me_now`)

Set these environment variables in production and use a long, unique passphrase.

### Accessing the admin UI

When running in HTTP / SSE modes (e.g. `node mcpServer.js --streamable-http`), the same process also serves an admin UI:

- Visit `http://localhost:3001/admin/login` (or your deployed base URL) to log in.
- On success you are redirected to `/admin/settings`.

### Workflow & API spec

On `/admin/settings`:

- **Workflow tool name**: Human label, used as a prefix in generated tool names (e.g. `airtable_list_records`).
- **API documentation URL**: Absolute URL to an OpenAPI 3.x JSON or YAML document.

Click **Save basics** to persist these settings.

### Authentication

You can configure how the MCP server authenticates to the target API:

- **API key / token**
  - `Header/query name`: header or query parameter name.
  - `Location`: `header`, `query`, or `bearer` (for `Authorization: Bearer` tokens).
  - `API key or token`: the secret; only a masked preview is ever returned from the API.

- **OAuth 2.0**
  - `Client ID`, `Client secret`
  - `Authorization URL`, `Token URL`
  - `Scopes` (space‑separated)
  - Click **Start OAuth consent** to open the provider’s login/consent page in a new tab.
  - On success you are redirected back to `/admin/settings` and tokens are stored on disk.

Tokens are kept only in `config/server-config.json` and never fully exposed via APIs.

### Endpoint discovery and selection

After basics and auth are saved:

1. Click **Discover endpoints** to fetch and parse the OpenAPI spec.
2. The UI lists all discovered endpoints (method, path, summary).
3. Select which endpoints should become MCP tools using the checkboxes.
4. Click **Save selection** to persist your choices.

Endpoints and their `selected` state are written to `config/server-config.json`.

### Dynamic MCP tools

At runtime:

- The server loads `config/server-config.json`.
- If a workflow `apiDocsUrl` is set and there are selected endpoints, it:
  - Fetches and parses the OpenAPI spec.
  - Builds MCP tools dynamically from the selected endpoints.
  - Injects auth (API key or OAuth bearer token) into outgoing requests.
- If no dynamic configuration is present yet, it falls back to the statically generated Postman tools under `tools/`.

Each generated tool:

- Has a name like `<workflow>_<operationId or method_path>`.
- Uses a JSON schema built from path/query parameters and (optionally) a `body` field for JSON request bodies.
- Performs a real HTTP call against the API using Node’s `fetch`.

When you change endpoint selection or auth, restart the Node process or reconnect MCP clients so they pick up the new tool list.

For production deployments, you can use Docker:

**1. Build Docker image**

The project comes bundled with the following minimal Docker setup:

```dockerfile
FROM node:22.12-alpine AS builder

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm install

COPY . .

ENTRYPOINT ["node", "mcpServer.js"]
```

### 🚢 Deploying with Dokploy / DigitalOcean

For Dokploy (or similar platforms on DigitalOcean):

- **Build & push image**: build this image and push it to your container registry (e.g. Docker Hub or DigitalOcean Container Registry)
- **Create a service** in Dokploy using that image
- **Environment**: configure `.env`‑equivalent variables (e.g. Airtable personal access token, base URL) in Dokploy’s environment settings
- **Command**: run `node mcpServer.js --streamable-http` (or use the default entrypoint and set `CMD`/args in Dokploy) so clients can call the `/mcp` HTTP endpoint
- **Port**: expose `PORT` (default `3001`) and terminate TLS at your proxy / load balancer

Repeat this pattern per‑platform (one container per MCP server) to build an enterprise‑grade catalogue of fully controlled MCP servers.

#### 🌐 Streamable HTTP

To run the server with Streamable HTTP support, use the `--streamable-http` flag. This launches the server with the `/mcp` endpoint enabled:

```sh
node mcpServer.js --streamable-http
```

#### 🌐 Server-Sent Events (SSE)

To run the server with Server-Sent Events (SSE) support, use the `--sse` flag. This launches the server with the `/sse` and `/messages` endpoints enabled:

```sh
node mcpServer.js --sse
```

#### 🖥️ Stdio (Standard Input/Output)

To run the server using standard input/output (stdio), simply run the script without any flags. This mode is ideal for CLI tools or programmatic integration via stdin and stdout.

```sh
node mcpServer.js
```
